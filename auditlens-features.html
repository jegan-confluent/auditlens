The Problem It Solves

Why raw Confluent Cloud audit logs are not enough

A moderately active Confluent Cloud organisation produces millions of audit events every day. The vast majority — easily 5.8M of a typical 5.87M daily total — are Kafka produce and fetch calls, mds.Authorize checks, and schema-registry authentication handshakes: background noise generated by every client connection, every message produced, every schema lookup. The real signal — the 73K events that include topic deletions, ACL changes, failed authentications, API key creation, and access denials — is buried underneath. A security or operations engineer reading raw audit logs must either scan millions of lines manually or build their own classification and filtering layer from scratch.

AuditLens is that classification and filtering layer. It reads the audit log topic in real time, classifies every event into one of four signal tiers (action_required, attention, informational, noise), enriches the actor field from a raw principal ID like sa-8nwyn7 to a human-readable name, and persists everything in a queryable database with a dashboard on top. Engineers stop reading logs and start triaging decisions.

How It Works

Five components, one pipeline

Confluent Cloud
confluent-audit-log-events topic
→
Forwarder
Classifies, enriches, deduplicates, writes to DB + Kafka topics
→
PostgreSQL
Two-table split: audit_events + audit_events_noise
→
FastAPI
REST API — events, summary, filters, system, admin
→
Next.js
Dashboard, Events, Resources, System, Settings pages

The forwarder assigns every event a signal_type using a deterministic rule cascade: events from internal platform topics or routine auth methods are noise; read-only lookups are informational; access and config changes are attention; failures, denials, destructive changes, and security-sensitive operations are action_required. Noise events are short-circuited at the consumer thread — they bypass the full enrichment pipeline and are written directly to audit_events_noise, saving roughly 83% of processor work. Signal events go through the complete enrichment chain before landing in audit_events.

Features

76 shipped features across 10 categories. Green = always-on. Amber = opt-in.

A
Ingestion & Forwarding
8 features
Kafka consumer with consumer group offsets always-on
Tracks exactly where it left off in the audit topic — restart the forwarder and it picks up from the last committed offset, with no events lost or replayed unnecessarily.
Bulk-noise short circuit always-on
Routes ~83% of events (mds.Authorize, kafka.Fetch, kafka.Produce, etc.) directly to the noise writer, bypassing enrichment entirely — keeps the signal pipeline fast under full Confluent Cloud load.
Two-table storage split always-on
Noise events land in a lean 9-column table (audit_events_noise); signal events land in the full-featured audit_events table — so signal queries never scan noise rows.
Multi-topic Kafka routing always-on
Publishes classified events to dedicated topics (audit.raw.v1, audit.enriched.v1, audit.signals.*, audit.dlq.v1) so downstream consumers can subscribe to only what they need.
Event fingerprinting & dedup always-on
SHA-based fingerprint per event prevents duplicate rows on consumer replay — safe to restart the forwarder at any time without creating duplicate entries in the DB.
Priority queue lanes always-on
Separate write queues for critical signals, normal events, bulk noise, and catalog updates — critical events reach the DB first even when the bulk queue is backed up.
Resource snapshot at ingest always-on
Extracts resource type, name, cluster, environment, blast radius hint, and production hint from every event payload at write time — no re-parsing needed for the dashboard or export.
Auto-migration on container start always-on
Alembic migrations run automatically when the API container starts — upgrade the stack with docker compose up -d and schema changes apply without manual intervention.
B
Signal Classification
8 features
Four-tier signal classification always-on
Every event is automatically sorted into action_required, attention, informational, or noise — operators see a triage queue, not a log stream.
Signal reason codes always-on
Each signal explains exactly why it was flagged — denied_access, destructive_change, failure_detected, config_changed, etc. — so operators don't need to reverse-engineer the classification decision.
Internal Confluent topic suppression always-on
Auto-created DLQ topics (error-lcc-*), internal metrics topics (_confluent*), and consumer group offset topics (__consumer_*) are classified as noise before any other rule — they never inflate action_required counts.
Confluent platform automation detection always-on
Identifies Confluent's own service accounts and classifies their routine activity as informational — so platform health-checks don't pollute the operator's signal feed.
Schema Registry failure reclassification always-on
Schema incompatibility failures (RegisterSchema errors) are downgraded from action_required to attention — recognising that schema evolution failures need a developer fix, not a security response.
Recommended action per event always-on
Every classified event carries a machine-readable recommendation ("Investigate immediately", "Confirm this was approved", "Verify owner and change window") — feeds notification payloads and the event detail drawer.
Plain-English event title & summary always-on
Generates a human-readable title and one-sentence summary for every event from method name and resource context — the dashboard shows what happened without requiring operators to know Confluent API method names.
Data / control / management plane tagging always-on
Tags every event as data_plane (produce/consume), control_plane (cluster/topic management), or management_plane (IAM/RBAC) — enables plane-level filtering in the Events page.

Configurable per-tier retention always-on
Set separate retention windows for signal events (default 7d), raw payloads (default 7d), and noise events (default 3d) — keep the security trail longer while discarding the noise faster to control disk usage.

Security
9 features
Bearer token auth with three roles opt-in
viewer can read events; responder can triage; admin can access cleanup and backfill endpoints — provide minimum-privilege tokens to each team member or service.
AES-256-GCM secret encryption always-on
All secrets stored in the database (webhook URLs, SR credentials, cold storage keys) are encrypted at rest and never returned in plaintext from any API endpoint — even admin endpoints return masked values.
HMAC constant-time token comparison always-on
Token validation uses hmac.compare_digest — prevents timing side-channel attacks where an attacker could guess token bytes by measuring response time differences.
CSP and security headers always-on
Content-Security-Policy, X-Frame-Options, Referrer-Policy, and X-Content-Type-Options are set on every API response — baseline hardening against clickjacking, MIME sniffing, and script injection.
Per-IP rate limiting always-on
All endpoints are rate-limited per client IP via slowapi — prevents a single misconfigured client from drowning the API under a flood of requests.
Ports bound to 127.0.0.1 by default always-on
All container ports (API, frontend, metrics, Postgres, Grafana) are bound to localhost only in the default compose — traffic cannot reach them from any other machine without an explicit port rebinding.
Forwarder runs read-only with dropped capabilities always-on
The forwarder container runs with read_only: true and cap_drop: ALL — a compromised forwarder process cannot write to the filesystem or call privileged kernel operations.
TLS via Caddy (production) opt-in
The production compose includes Caddy as a reverse proxy that handles ACME certificate issuance and renewal automatically — set AUDITLENS_DOMAIN and HTTPS just works.
No telemetry or phone-home always-on
The only outbound connections AuditLens makes are to your Confluent Kafka endpoint (always) and to api.confluent.cloud (only when IAM_ENRICHMENT_ENABLED=true) — your audit data stays in your infrastructure.

Signal Types Explained

Every event is assigned exactly one signal_type. Here is what each means and what the operator should do.

noise
~83% of all events
What triggers it
  • mds.Authorize — every RBAC permission check
  • kafka.Fetch, kafka.Produce — client I/O
  • schema-registry.Authentication — handshakes
  • Any event on internal platform topics (_confluent*, error-lcc-*)
  • Confluent platform automation service accounts (routine activity)
What to do: nothing — stored separately, hidden by default
informational
~15% of all events
What triggers it
  • Any Get* or List* method — read-only lookups
  • Successful authentications that aren't flagged denials
  • Operational events (constructive or operational impact types)
  • Tableflow GetTable / ListTables helpers
What to do: monitor — part of the audit trail, no immediate action
attention
~1.5% of all events
What triggers it
  • ACL / RBAC changes — grant, revoke, rolebinding
  • API key creation or deletion
  • Configuration changes — alterconfigs, updateconfig
  • Topic, schema, cluster, or connector creates/updates
  • Schema Registry RegisterSchema failures (schema evolution)
What to do: review — verify the owner and change window
action_required
<0.5% of all events
What triggers it
  • Any denied access attempt
  • Delete, destroy, drop, terminate operations
  • Failed authentication (kafka.authentication, ksql.authenticate)
  • Any API failure on a destructive or security-sensitive operation
  • Access-transparency events with security-sensitive impact
What to do: act now — investigate immediately

Quick Start

From clone to running dashboard in under 5 minutes

1
Prerequisites
You need Docker Desktop (≥6 GB RAM allocated), Python 3.11+, and access to your Confluent Cloud audit log topic.
# Confluent Cloud credentials you will need:
#   - Audit log Kafka bootstrap endpoint
#     (e.g. pkc-abc123.us-east-1.aws.confluent.cloud:9092)
#   - Kafka API Key + Secret with READ access to confluent-audit-log-events
#   - Destination Kafka bootstrap + API Key + Secret (for enriched topics)
#
# Optional (enables actor display names):
#   - CONFLUENT_CLOUD_API_KEY + CONFLUENT_CLOUD_API_SECRET
2
Clone and run the setup wizard
The wizard collects credentials, validates Kafka connectivity, generates .env and .secrets, and starts all services.
git clone <repo-url>
cd AuditLens
./setup
When complete you will see:
✅  AuditLens is ready.
    Open http://localhost:3000  (local dev only)
3
Open the dashboard
The Dashboard page shows signal counts, top active actors, and the event volume chart. The tables will populate as the forwarder consumes events from your audit topic — usually within a few seconds on a live cluster.
# Dashboard:            http://localhost:3000        (local dev only)
# API:                  http://127.0.0.1:8080        (local dev only)
# Forwarder health:     http://127.0.0.1:8003/health (local dev only)
# Grafana:              http://127.0.0.1:3001        (local dev only)

# Common commands after setup:
make status     # Container health + API check
make logs       # Tail forwarder logs
make stop       # Stop all services
make test       # Run the test suite (701 tests)
No Kafka? SQLite demo mode — seeds sample data, no credentials needed:
scripts/run_sqlite_demo.sh
# Open http://127.0.0.1:3000  (local dev only)
